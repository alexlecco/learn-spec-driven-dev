Spec: Hero Section 
SPEC: Hero Section
Goal: Immediately communicate what SDD is and create momentum for the user
      to start learning — delivering value within 10 seconds of landing.
Users: First-time visitors, primarily mid/senior developers
Acceptance criteria:
- AC1: Viewport-height section with aurora/mesh CSS background
- AC2: Headline "Write the spec. Build the thing. Repeat." in Syne font
- AC3: Two CTAs: "Start learning" (→ #learn) and "See how this was built" (→ #meta)
- AC4: Animated spec snippet cycles through 3 examples with typewriter effect
- AC5: Scroll badge fades in after 1.5s
Out of scope:
- Video backgrounds
- User authentication
Spec: Concept Section 
SPEC: Conceptual Introduction
Goal: Define SDD clearly enough that Marco can explain it to a colleague
      in one sentence after reading this section.
Users: First-time visitors unfamiliar with SDD
Acceptance criteria:
- AC1: Three-card bento layout (The Problem, The Spec, The Loop)
- AC2: Each card has icon, title, 2-sentence description, and meta-spec block
- AC3: Horizontal comparison strip "Ad-hoc dev vs SDD" with 5 dimensions
- AC4: Section stacks to 1 column on mobile
Out of scope:
- Comparison to BDD/TDD/DDD methodologies
Spec: Learning Path Section 
SPEC: 5-Module Learning Path
Goal: Guide Marco through the full SDD methodology in a self-paced,
      progress-tracked format so he retains the key concepts.
Users: Mid/senior developers learning SDD for team adoption
Acceptance criteria:
- AC1: Progress bar reflects localStorage completion state
- AC2: Five module cards in asymmetric bento grid
- AC3: Each card: number, title, description, time estimate, CTA
- AC4: Module content expands inline via accordion (no page navigation)
- AC5: "Mark as complete" persists to localStorage
Out of scope:
- Server-side progress tracking
- Assessments or quizzes
Spec: Spec Playground 
SPEC: Interactive Spec Playground
Goal: Let Marco write a real spec and get instant structured feedback so he
      leaves with a validated spec he can use in his next ticket.
Users: Developers actively learning or practicing SDD
Acceptance criteria:
- AC1: Two-pane layout (editor left, analysis right) on desktop; stacked on mobile
- AC2: Textarea editor with line numbers, tab-to-indent, Ctrl+Enter shortcut
- AC3: Validation engine scores 4 dimensions: structure, AC quality,
       goal clarity, scope definition
- AC4: Score displayed as circular CSS conic-gradient meter
- AC5: Three example specs loadable via buttons
- AC6: "Copy this spec" copies to clipboard
Out of scope:
- Saving specs to a server
- Sharing specs via URL
Spec: Meta-Case Section 
SPEC: Meta-Case "How This Site Was Built"
Goal: Demonstrate SDD credibility by showing every phase's spec, decisions,
      and retrospective for the very site the user is reading.
Users: Skeptical developers who want proof that SDD works in practice
Acceptance criteria:
- AC1: Vertical CSS timeline showing all 7 phases with real git commit hashes
- AC2: "Inspect spec" floating button reveals spec of any section on click
- AC3: Build log reader renders /docs/build-log.md with basic markdown support
- AC4: All section meta-spec blocks are filled with actual specs
Out of scope:
- Real-time git integration
- External analytics
