What is SDD? / ¿Qué es SDD?

Specs before code. Always. El spec primero. Siempre.

Stop building the wrong thing perfectly. Write the spec first. Dejá de construir perfectamente la cosa equivocada. Escribí el spec primero.

The Problem El Problema

Developers ship the wrong thing not because they can't code, but because the goal was never written down. Requirements live in Slack threads, hallway conversations, and half-remembered meetings. Los desarrolladores construyen lo incorrecto no porque no sepan codear, sino porque el objetivo nunca se escribió. Los requisitos viven en hilos de Slack, conversaciones de pasillo y reuniones medio recordadas.

The Spec El Spec

A spec is a structured short document: the goal, the users, the acceptance criteria, and what's out of scope. Written before a single line of code. It fits in a comment block, a ticket, or a Notion page. Un spec es un documento corto y estructurado: el objetivo, los usuarios, los criterios de aceptación y qué está fuera del alcance. Se escribe antes de una sola línea de código. Cabe en un bloque de comentario, un ticket o una página de Notion.

The Loop El Ciclo

After building, you write a brief retrospective: what matched the spec, what diverged, and why. That retro feeds the next spec. The loop keeps your process honest and your team aligned over time. Después de construir, escribís una retrospectiva breve: qué coincidió con el spec, qué divergió y por qué. Esa retro alimenta el próximo spec. El ciclo mantiene tu proceso honesto y tu equipo alineado con el tiempo.

Anatomy of a spec Anatomía de un spec

What fields every spec must have. A live annotated example with highlighted regions you can explore. Qué campos debe tener cada spec. Un ejemplo anotado en vivo con regiones destacadas para explorar.
5 min read

The 5 required fields Los 5 campos obligatorios

Every spec must answer five questions before a line of code is written: Cada spec debe responder cinco preguntas antes de escribir una línea de código:
  • SPEC: — The feature name. Short, memorable, searchable. / El nombre de la funcionalidad. Corto, memorable, buscable.
  • Goal: — What outcome the user achieves. Starts with "Allow", "Enable", "Let". / Qué resultado logra el usuario. Comienza con "Permitir", "Habilitar", "Dejar".
  • Users: — Who uses this, on what device/context. / Quién lo usa, en qué dispositivo/contexto.
  • Acceptance criteria: — Testable, verb-first statements. Each AC is a test you can run. / Declaraciones verificables, con verbo al inicio. Cada AC es una prueba que podés ejecutar.
  • Out of scope: — Explicit list of what this feature does NOT include. / Lista explícita de lo que esta funcionalidad NO incluye.

Annotated example Ejemplo anotado

SPEC: Password reset Goal: Allow registered users to reset their password via email so they can regain access without contacting support. Users: Registered users on web and mobile Acceptance criteria: - Request reset → email sent within 60 seconds - Reset link expires after 24 hours - New password must be 8+ chars, 1 uppercase, 1 number - Old password is invalidated immediately on reset Out of scope: - SMS-based reset - Admin-initiated resets
Notice: the Goal uses "so that" to connect the action to the user outcome. The ACs are all testable. Out of scope is specific, not vague ("etc." is never acceptable). Notá: el Goal usa "para que" para conectar la acción con el resultado del usuario. Los ACs son todos verificables. El Out of scope es específico, no vago ("etc." nunca es aceptable).

02

Writing acceptance criteria Escribir criterios de aceptación

The difference between vague and precise ACs. Side-by-side examples with interactive quality check. La diferencia entre ACs vagos y precisos. Ejemplos comparativos con verificación interactiva de calidad.
7 min read

What makes a good AC? ¿Qué hace a un buen AC?

A good acceptance criterion is: testable (can I run a test that passes or fails?), verb-first (starts with an action), and unambiguous (no vague qualifiers like "fast", "nice", "works well"). Un buen criterio de aceptación es: verificable (¿puedo ejecutar un test que pase o falle?), con verbo al inicio (comienza con una acción) y sin ambigüedad (sin calificadores vagos como "rápido", "bonito", "funciona bien").

Bad AC vs Good AC AC malo vs AC bueno

✗ Vague / Vago

The login should work well and be fast.
Users can edit their profile.
Error messages are nice.

✓ Precise / Preciso

Login responds in <500ms for 95th percentile.
Submit profile form → changes saved, toast shown.
Invalid email → red border + "Invalid email" text.

The vague word blacklist La lista negra de palabras vagas

If any of these words appear in an AC, rewrite it: works, good, nice, easy, fast, smooth, better, improved, simple, intuitive, user-friendly. Replace each with a measurable threshold or an observable outcome. Si alguna de estas palabras aparece en un AC, reescribilo: funciona, bueno, bonito, fácil, rápido, suave, mejor, mejorado, simple, intuitivo, amigable. Reemplazá cada una con un umbral medible o un resultado observable.

03

Spec granularity Granularidad del spec

How detailed should a spec be? Navigate the spectrum from too vague to too rigid. ¿Qué tan detallado debe ser un spec? Navegá el espectro de demasiado vago a demasiado rígido.
6 min read

The granularity spectrum El espectro de granularidad

Specs that are too vague leave too much to interpretation. Specs that are too detailed become implementation prescriptions that prevent developer judgment. The sweet spot: enough detail to remove ambiguity about what to build, not how to build it. Los specs demasiado vagos dejan demasiado a la interpretación. Los specs demasiado detallados se convierten en prescripciones de implementación que impiden el juicio del desarrollador. El punto ideal: suficiente detalle para eliminar la ambigüedad sobre qué construir, no cómo construirlo.
Too vague / Muy vago Just right / Justo Too rigid / Muy rígido
"Build a search feature"
"Search returns results matching all query terms, sorted by relevance, ≤200ms"
"Use Elasticsearch 8.2 with BM25 scoring, index refreshed every 30s, use a debounced React hook at 250ms"

Rules of thumb Reglas prácticas

  • Spec the WHAT (behavior), not the HOW (implementation). Especificá el QUÉ (comportamiento), no el CÓMO (implementación).
  • If two developers would build different things from this spec, add a clarifying AC. Si dos desarrolladores construirían cosas distintas con este spec, agregá un AC aclaratorio.
  • If a developer would have no creative latitude, remove an AC. Si un desarrollador no tendría latitud creativa, eliminá un AC.

04

The retrospective loop El ciclo de retrospectiva

After building, how to write a brief retro that feeds the next spec. Uses this site's own build log as a real example. Después de construir, cómo escribir una retro breve que alimenta el próximo spec. Usa el build log de este sitio como ejemplo real.
5 min read

Why write a retro? ¿Por qué escribir una retro?

Without a retro, every feature is a one-way door: you build it, ship it, and forget what you learned. A retro closes the loop: it captures what the spec got right, what it missed, and what to do differently next time — turning one-time decisions into team knowledge. Sin una retro, cada funcionalidad es una puerta de un solo sentido: la construís, la enviás y olvidás lo que aprendiste. Una retro cierra el ciclo: captura qué acertó el spec, qué se perdió y qué hacer diferente la próxima vez, convirtiendo decisiones únicas en conocimiento del equipo.

The 3-question retro format El formato de retro de 3 preguntas

  • What matched the spec? — Celebrate what the spec clarified correctly. Reinforce it. ¿Qué coincidió con el spec? — Celebrá lo que el spec clarificó correctamente. Reforzalo.
  • What diverged, and why? — Was the spec wrong? Was reality different? Who decided to deviate? ¿Qué divergió y por qué? — ¿Estaba equivocado el spec? ¿Era diferente la realidad? ¿Quién decidió desviarse?
  • What would you spec differently? — One concrete change to the spec format or content for next time. ¿Qué especificarías diferente? — Un cambio concreto al formato o contenido del spec para la próxima vez.

Real example: Phase 1 of this site Ejemplo real: Fase 1 de este sitio

What matched: Single-page architecture, token-based CSS, no build tool. What diverged: README "Lessons learned" section was written before lessons were learned. Required retroactive invention. What we'd spec differently: Leave "Lessons learned" as placeholder; fill in after Phase 7 with genuine observations.

Team adoption playbook Playbook de adopción en equipo

Six concrete steps to introduce SDD to a team. Week-by-week rollout. Downloadable checklist. Seis pasos concretos para introducir SDD en un equipo. Rollout semana a semana. Checklist descargable.
8 min read

The 6-step adoption timeline El cronograma de adopción de 6 pasos

Week 1
Solo pilot. Write specs for your next 3 tickets — alone. Don't announce it yet. Observe how it changes your thinking before building. Piloto individual. Escribí specs para tus próximos 3 tickets — solo. No lo anunciés todavía. Observá cómo cambia tu forma de pensar antes de construir.
Week 2
Show one example. In a 1:1 or standup, show a spec you wrote and the ticket it replaced. Let the quality speak, don't pitch the methodology. Mostrá un ejemplo. En un 1:1 o standup, mostrá un spec que escribiste y el ticket que reemplazó. Dejá que la calidad hable, no presentes la metodología.
Week 3
Pair write. Write a spec together with one teammate for an upcoming feature. Ask for their corrections. The spec becomes shared understanding, not your document. Escribí en pareja. Escribí un spec junto con un compañero para una funcionalidad próxima. Pedí sus correcciones. El spec se convierte en entendimiento compartido, no en tu documento.
Week 4
Propose the template. Suggest adding the spec format to your ticket template in Jira/Linear/GitHub Issues. Frame it as "we added 5 fields that prevent rework" — not "a new process". Proponé el template. Sugería agregar el formato de spec al template de tickets en Jira/Linear/GitHub Issues. Enmarcalo como "agregamos 5 campos que previenen retrabajo", no como "un nuevo proceso".
Week 5–6
First team retro. After the first feature built with a shared spec ships, run a 15-minute retro using the 3-question format. Document it in the ticket or a shared doc. This creates institutional memory. Primera retro del equipo. Después de que se envíe la primera funcionalidad construida con un spec compartido, hacé una retro de 15 minutos usando el formato de 3 preguntas. Documentala en el ticket o un doc compartido. Esto crea memoria institucional.
Week 7+
Make it a norm. At this point, "can you add the spec fields to this ticket?" becomes a normal code-review comment. No more pitching — it's just how your team works. Hacelo la norma. En este punto, "¿podés agregar los campos del spec a este ticket?" se convierte en un comentario normal de revisión de código. Sin más presentaciones — así es como trabaja tu equipo.
Download the checklist — a markdown file with the 6 steps formatted as a todo list you can paste into your team's wiki, Notion, or GitHub README. Descargá el checklist — un archivo markdown con los 6 pasos formateados como lista de tareas que podés pegar en la wiki, Notion o README de tu equipo.
